# 外部系统对接方案 — Python 仿真引擎接入综合仿真平台

> 本文档基于《轨交多系统平台接口协议汇总20260630》整理，明确 Python 自研仿真引擎 `sim_engine/` 替换原有 RT-LAB 车辆模型接入外部平台时，需要对接的设备、通信协议、数据流及实施计划。

---

## 一、对接目标

将 Python 自研仿真引擎（`sim_engine/`）接入综合仿真平台，替换原有的 RT-LAB 实时车辆模型，实现：

1. **精确停车**：利用引擎内部的三段式/ATO 逻辑独立完成站台对标停车 ±1m
2. **真实外设接入**：接收真实司机台 PLC 手柄输入，驱动网络屏和信号屏显示，响应信号系统状态
3. **数据共享**：通过总控节点向信号系统提供车辆状态数据

### 2.2 数据流方向

```
PLC (手柄/按钮) ──TCP 100ms──▶ 你的引擎(决策) ──TCP 572B──▶ 网络屏(显示)
                                      │         ──TCP 66B───▶ 信号屏(显示)
                                      │
                                      ├──UDP 20ms──▶ 总控节点 ──▶ 信号系统(监控)
                                      │
                                      ◀──UDP 100ms── 总控节点 ── 信号系统状态
```

## 四、通信协议详述

### 4.1 总控节点 UDP 20ms — 高频数据交换

#### 4.1.1 你 → 总控节点（每 20ms 发送）

| 序号 | 数据类型 | 字节数 | 字段说明 | 来源（你的引擎） |
|------|---------|--------|---------|----------------|
| 1 | double | 8 | 列车1 加速度 (m/s²) | `dynamics.py` 解算结果 |
| 2 | double | 8 | 列车1 速度 (m/s) | `train.speed` 换算 |
| 3 | double | 8 | 列车1 累计里程 (m) | `train.position` |
| 4 | double | 8 | 列车2 加速度 | |
| 5 | double | 8 | 列车2 速度 | |
| 6 | double | 8 | 列车2 累计里程 | |
| ... | ... | ... | ... | |
| 58 | double | 8 | 列车20 加速度 | |
| 59 | double | 8 | 列车20 速度 | |
| 60 | double | 8 | 列车20 累计里程 | |

**总长：** 20 列车 × 3 字段 × 8 字节 = **480 字节**

**用途：** 总控节点收到后转发给信号系统，信号系统据此获取你的位置和速度，用于安全监控。

#### 4.1.2 总控节点 → 你（每 20ms 接收）

| 序号 | 数据类型 | 字节数 | 字段说明 | 处理方式 |
|------|---------|--------|---------|---------|
| 1 | double | 8 | 列车1 指令（1=加速, 2=减速, 0=惰行） | **你的引擎不执行，仅供参考** |
| 2 | double | 8 | 列车1 加减速百分比 (0~100) | **你的引擎忽略，用自己的牵引特性曲线算** |
| 3 | double | 8 | 列车2 指令 | 同上 |
| 4 | double | 8 | 列车2 加减速百分比 | 同上 |
| ... | ... | ... | ... | |
| 39 | double | 8 | 列车20 指令 | |
| 40 | double | 8 | 列车20 加减速百分比 | |

**总长：** 20 列车 × 2 字段 × 8 字节 = **320 字节**

**关键设计决策：** 你的引擎保留内部三段式/ATO 决策逻辑，外部指令仅作为参考信息（如检测紧急制动信号），**不替代你的精确停车算法**。

### 4.2 总控节点 API 500ms — 参数配置（需确认替代方案）

原协议使用 RT-LAB 私有 API（`OpalOpenProject()` 等），你的 Python 引擎无法调用。需与平台方协商替代方案，建议使用 **TCP 或 HTTP 接口**。

**你 → 总控节点（每 500ms）：**

| 序号 | 数据类型 | 说明 |
|------|---------|------|
| 1 | float | 列车1 编号 |
| 2 | float | 列车1 激活端（1/2） |
| 3 | float | 列车1 方向（上行/下行） |
| 4 | float | 列车1 加速度 |
| 5 | float | 列车1 速度 |
| 6 | float | 列车1 累计里程 |
| ... | ... | ...（最多 20 列车） |

**总控节点 → 你（每 500ms）：**

| 序号 | 数据类型 | 说明 |
|------|---------|------|
| 1 | float | 列车1 编号 |
| 2 | float | 列车1 操作指令 |
| 3 | float | **列车1 seg号**（外部平台区段编号） |
| 4 | float | **列车1 偏移**（外部平台位置偏移量） |
| 5 | float | 列车1 方向 |
| 6 | float | 列车1 激活端 |
| ... | ... | ...（最多 20 列车） |

**重点：** 需要建立 **seg号+偏移 → 公里标** 的映射关系，将外部平台的位置表达转换为你的引擎内部公里标。

## 五、你的引擎决策逻辑

### 5.1 控制模式切换

```
PLC 每 100ms 发来手柄/按钮状态
    │
    ├── ATO 启动按钮=已按下 AND 钥匙开关=锁定
    │   └── ▶ 自动模式（ATO/三段式）
    │       ├── 你的 three_stage.py 算牵引/制动
    │       ├── 制动点：S_brake = v²/(2a) × safety_factor
    │       └── 目标：站台中心 ±1m 停车
    │
    └── ATO 启动按钮=未按下 OR 钥匙开关=解除
        └── ▶ 手动模式
            ├── 读取 PLC 主手柄状态+牵引极位/制动极位
            ├── 通过 traction_curve 算牵引力
            └── dynamics.step() 执行
```

### 5.2 精确停车逻辑（不变）

你的引擎内部的三段式/ATO 精确停车逻辑**完全保留**：

```
每个区间循环执行：

1. 牵引阶段
   条件：v < v_target (v_target = 0.8 × speed_limit)
   动作：施加最大牵引力

2. 惰行阶段
   条件：已牵引至目标速度，且距离站台 > 制动触发距离
   动作：零牵引零制动

3. 制动阶段（前馈 + P 微调）
   条件：当前位置 + 制动触发距离 >= 站台中心
   动作：前馈 a=v²/2d + P 微调
   目标：v=0 且 |x - station_center| < 1m
```

### 5.3 外部输入的整合

```
外部信号系统（通过总控节点，UDP 20ms 指令）：
  · 信号机红灯 → 你的引擎提前制动/禁止发车
  · 道岔反位 → 限速运行（侧向限速）
  · 前车距离 < 安全距离 → ATP 紧急制动

PLC 输入（TCP 100ms）：
  · 紧急制动按钮锁定 → 立即触发紧急制动
  · 主手柄状态 → 手动模式下决定牵引/制动
  · 方向手柄 → 决定运行方向
  · ATO 启动按钮 → 切换自动模式
```

### 5.4 对外输出数据流

```
每 20ms → 总控节点（UDP 480 字节）：
  加速度、速度、累计里程 → 信号系统监控

每 100ms → PLC（TCP 26 字节）：
  指示灯状态、速度回显、模式标识

按需 → 网络屏（TCP 572 字节）：
  速度、加速度、牵引力、网压、限速、级位、模式
  门状态、制动状态、站信息、蓄电池信息…

按需 → 信号屏（TCP 66 字节）：
  速度、加速度、限速、模式、距下一站距离
  牵引/制动/紧急制动状态、站信息
```

## 八、实施计划

### 8.1 前提条件（需与平台方确认）

| 序号 | 确认事项 | 影响 |
|------|---------|------|
| 1 | 总控节点/PLC/网络屏/信号屏的 IP 和端口是否与文档一致 | 如不同需调整配置 |
| 2 | API 500ms 通道的替代方案（RT-LAB API 不能用） | 决定 TCP 或 HTTP 方案 |
| 3 | 线路数据（seg+offset 映射表、坡度/曲率/限速/车站位置） | 录入你的引擎 |
| 4 | 信号系统发给总控的数据格式是否与文档一致 | 确认解析逻辑 |
| 5 | 网络屏和信号屏的 TCP 连接方式（是否允许你主动连） | 确认网络策略 |

### 8.2 开发计划

| 阶段 | 工作内容 | 预估时间 |
|------|---------|---------|
| **1. 总控 UDP 通信** | 实现 UDP 20ms 收发 + 编解码 | 1 天 |
| **2. PLC 协议** | 实现 TCP 客户端 + 46B/26B 报文解析 | 1.5 天 |
| **3. 网络屏/信号屏** | 实现 TCP 572B/66B 报文组装发送 | 1 天 |
| **4. 桥接层** | 编排器新增 external 模式，整合所有外部输入 | 1 天 |
| **5. 数据录入** | 录入车辆参数 + 线路数据 + 坐标映射 | 0.5 天 |
| **6. 联调测试** | 与所有外设联调，验证闭环 | 2-3 天 |
| **合计** | | **7-9 天** |

### 8.3 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| UDP 丢包 | 信号系统收不到你的位置 | 超时回退到内部模式 |
| 外部指令与内部逻辑冲突 | 控车异常 | 以内部逻辑为主，外部指令仅参考 |
| 坐标映射不一致 | 位置偏差 | 联调时逐点验证 seg+offset → 公里标 |
| RT-LAB API 无替代方案 | 500ms 通道缺失 | 提供 TCP 接口作为替代 |
| PLC 报文格式与文档有出入 | 解析错误 | 联调时用捕包工具验证 |
| 三路 TCP 同时连接 | 连接管理复杂 | 统一在 bridge 中管理生命周期 |
