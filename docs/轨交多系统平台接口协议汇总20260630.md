## （一）信号系统 

### 列车运行控制 

#### 1 引言 
**1.1 目的** 
本文档是为了描述信号系统与总控数据库节点的接口，作为子系统需求文档的输入文件，指导后续的研发和测试工作。本接口说明书不包括ATS与总控的接口。 

**1.2 范围** 
该接口文档适用于仿真平台项目。 

**1.3 术语和定义** 

| 缩写 | 中文 |
|---|---|
| CBTC | 基于通信的列车运行控制  |
| IP | 互联网络协议  |
| UDP | 面向无连接的传输控制协议  |
| TBD | 有待定义  |

#### 2 总体描述 
信号系统和车辆的动力学模型、司机台、仿真电气单元通过总控的数据库节点交互信息。信号方设置一个接口转换单元，与总控数据库节点通信。与车辆方均将信息发送至此节点，由节点分派应用信息，使用UDP协议。 

#### 3 接口详细描述 
**3.1 接口连接方式** 
**3.1.1 物理/电气接口** 
信号系统与总控数据库节点之间的硬件连接是使用标准的RJ45端子及双绞线物理连接，物理层特征如下表所示： 

表 1 : 物理层特征 

| 物理特性 | 选择 |
|---|---|
| | 双绞线  |
| 标准 | IEEE 802.3(Ethernet)  |
| 物理连接 | Ethernet 10/100BaseT  |
| 传输 | Full-duplex  |
| 编码顺序 | Big endian  |
| 带宽 | 100Mbits/s  |
| 编码方式 | Manchester  |

**3.1.2 静态描述** 
信号系统的传输数据都接入信号接口转换单元，转换单元与总控数据库节点分别连接信号系统的冗余网和总控调度以太网。信号网和总控网通过数据库节点隔离。 

**3.1.3 动态描述** 
信号接口转换单元周期性每隔250毫秒，向数据库服务器发送全线的道岔状态信息、仿真车的牵引制动状态和牵引制动百分比信息。数据库服务器转发给实时动力学模型、一维动力学模型和总控驾驶界面进行数据处理。当进行加车、删车和改车操作时，线路仿真实时地将列车信息发送至动力学模型。 
总控数据库节点每隔100ms向信号接口转换单元转发实时动力学模型、一维动力学模型、总控驾驶界面的数据信息。 

**3.1.4 通信故障处理** 
若发生通信故障，可通过捕包软件捕包查看是否接收到信号网或调度以太网的数据。如果有相应的线路仿真信息，则表明问题可能在动力学模型。如果有相应的动力学模型信息，则表明问题出现在线路仿真端。若双方发数据都正常，则表明问题可能出在数据库服务器端。 

**3.2 通信层次结构描述** 
本接口通信层次由物理层、数据链路层、网络层、传输层和应用层组成，接口通信层次关系描述如下图所示： 
*(图3 : 线路仿真-动力学模型接口层次描述)* 

* **3.2.1 物理层** ：总控数据库节点和信号接口转换单元采用RJ45接口。 
* **3.2.2 数据链路层** ：MAC(Medium Access Control)子层基于IEEE 802.3标准，每个设备都使用2个MAC地址。  MAC头由14个字节组成，1个帧校验序列(4字节)将被加在Ethernet帧后面。 
* **3.2.3 网络层** ：本接口使用IPv4协议作为网络层的协议。 
* **3.2.4 传输层** ：本接口使用UDP/IP协议作为传输协议。 
* **3.2.5 应用层** ：定义应用交换消息的数据格式，实现信号系统与总控数据库节点之间的信息交换。

线路的上行/下行方向由信号系统定义。 
Link编号由信号厂家给出，用于描述全线的线路拓扑。 
在仿真线路上加车时，规定列车1端默认朝向下行方向，2端默认朝向上行方向。操作人员通过开启不同端的司机室钥匙，得到期望的实际运行方向。

### ATP 
#### ATP通信协议规范 

**通信接口** 
ATP系统通过冗余以太网与DMI系统传输信息，具体参数如下： 

| 物理特性 | 双绞线 |
|---|---|
| 标准 | IEEE 802.3  |
| 物理连接 | Ethernet 10/100BaseT  |
| 传输 | Full-duplex  |
| 编码顺序 | Big Endian  |
| 带宽 | 100Mbits/s  |
| 编码方式 | Manchester  |

**数据存放大小端约定** 
本协议中所有多字节元素存放格式均为大端模式。 

**校验码** 
检验码为CRC48校验。 
CRC48校验的计算需要将帧长、发送序号、应用数据一起作为计算的一部分。 
CRC48多项式为：X42+X39+X35+X34+X32+X29+X27+X26+X21+X17+X16+X13+X7+X5+X3+X1+1。 [cite: 114, 115]

**动态交互** 
ATP系统周期(通信周期为160ms)向DMI发送信息，DMI接收到ATP数据时向ATP回复信息。*(图2 数据交互图)* [cite: 117, 118]

| | 发送序列号 | 上次接收对方序列号 | 应用数据包长度 | 应用数据包 | 校验码（CRC48） |
|---|---|---|---|---|---|
| 长度 | 4字节 | 4字节 | 2字节 | ATP->DMI 129字节 <br> DMI->ATP 40字节 | 6字节  |

**数据处理** 
数据发送时，在未接收到对方消息或者通信中断时，则[上次接收对方序列号]需填写为默认值（0xFFFFFFFF）。 
若接收序列号大于上次接收序列号则认为该数据为新数据，则认为有效。若接收序列号小于等于上次接收序列号，则认为该数据为冗余数据，直接丢弃。 
在通信中断后，需要重置上次接收序列号为默认值。 
冗余处理后，接收方在接收数据时需对数据时效进行检查。若时效超过最大时效时，直接丢弃该数据。 
参考延迟时间 = (接收方当前序列号-报文中[上次接收对方序号])*0.16。报文时效性约定为1.6s。 
接收到对方报文时，需对报文的合法性进行检查（例如值域，CRC）等，检查失败，则认为该周期未收到对方数据。 
双方在连续10个周期未接收到对方数据，则直接判断为双方通信中断。
