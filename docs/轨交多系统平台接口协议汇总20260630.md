# **（一）信号系统**

<a id="_Toc233726975"></a>
## **列车运行控制**

#### **1 引言**

<a id="_Toc6747"></a>
##### **1.1 目的**

本文档是为了描述信号系统与总控数据库节点的接口，作为子系统需求文档的输入文件，指导后续的研发和测试工作。本接口说明书不包括ATS与总控的接口。

<a id="_Toc14683"></a>
##### **1.2 范围**

该接口文档适用于仿真平台项目。

<a id="_Toc12686"></a>
##### **1.3 术语和定义**

<table><tr><td><p>缩写</p></td><td><p>中文</p></td></tr><tr><td><p>CBTC</p></td><td><p>基于通信的列车运行控制</p></td></tr><tr><td><p>IP</p></td><td><p>互联网络协议</p></td></tr><tr><td><p>UDP</p></td><td><p>面向无连接的传输控制协议</p></td></tr><tr><td><p>TBD</p></td><td><p>有待定义</p></td></tr></table>

<a id="_Toc14853"></a>
#### **2 总体描述**

信号系统和车辆的动力学模型、司机台、仿真电气单元通过总控的数据库节点交互信息。信号方设置一个接口转换单元，与总控数据库节点通信。与车辆方均将信息发送至此节点，由节点分派应用信息，使用UDP协议。

<a id="_Toc125"></a>
#### **3 接口详细描述**

<a id="_Toc18795"></a>
##### **3.1 接口连接方式**

<a id="_Toc9680"></a>
###### **3.1.1物理/电气接口**

信号系统与总控数据库节点之间的硬件连接是使用标准的RJ45端子及双绞线物理连接，物理层特征如下表所示：

<table><tr><td rowspan="2"><p>物理特性</p></td><td><p>选择</p></td></tr><tr><td><p>双绞线</p></td></tr><tr><td><p>标准</p></td><td><p>IEEE 802.3(Ethernet)</p></td></tr><tr><td><p>物理连接</p></td><td><p>Ethernet 10/100BaseT</p></td></tr><tr><td><p>传输</p></td><td><p>Full-duplex</p></td></tr><tr><td><p>编码顺序</p></td><td><p>Big endian</p></td></tr><tr><td><p>带宽</p></td><td><p>100Mbits/s</p></td></tr><tr><td><p>编码方式</p></td><td><p>Manchester</p></td></tr></table>
  
表 1 : 物理层特征

<a id="_Toc1665"></a>
###### **3.1.2 静态描述**

信号系统的传输数据都接入信号接口转换单元，转换单元与总控数据库节点分别连接信号系统的冗余网和总控调度以太网。信号网和总控网通过数据库节点隔离。

<a id="_Toc28798"></a>
###### **3.1.3动态描述**

信号接口转换单元周期性每隔250毫秒，向数据库服务器发送全线的道岔状态信息、仿真车的牵引制动状态和牵引制动百分比信息。数据库服务器转发给实时动力学模型、一维动力学模型和总控驾驶界面进行数据处理。当进行加车、删车和改车操作时，线路仿真实时地将列车信息发送至动力学模型。

总控数据库节点每隔100ms向信号接口转换单元转发实时动力学模型、一维动力学模型、总控驾驶界面的数据信息。

<a id="_Toc1349"></a>
###### **3.1.4通信故障处理**

若发生通信故障，可通过捕包软件捕包查看是否接收到信号网或调度以太网的数据。如果有相应的线路仿真信息，则表明问题可能在动力学模型。如果有相应的动力学模型信息，则表明问题出现在线路仿真端。若双方发数据都正常，则表明问题可能出在数据库服务器端。

<a id="_Toc10728"></a>
##### **3.2 通信层次结构描述**

本接口通信层次由物理层、数据链路层、网络层、传输层和应用层组成，接口通信层次关系描述如下图所示：

![](images/5e383d1d45b7de4b9fa9143e4e9ea41713e055a9d72ac00614059c40393f341c.jpg)  
图3 : 线路仿真-动力学模型接口层次描述

<a id="_Toc16773"></a>
###### **3.2.1 物理层**

总控数据库节点和信号接口转换单元采用RJ45接口。

<a id="_Toc7887"></a>
###### **3.2.2数据链路层**

MAC(Medium Access Control)子层基于IEEE 802.3标准，每个设备都使用2个MAC地址。

MAC头由14个字节组成，1个帧校验序列(4字节)将被加在Ethernet帧后面。

<a id="_Toc15490"></a>
###### **3.2.3网络层**

本接口使用IPv4协议作为网络层的协议。

<a id="_Toc7397"></a>
###### **3.2.4传输层**

本接口使用UDP/IP协议作为传输协议。

<a id="_Toc180"></a>
###### **3.2.5应用层**

定义应用交换消息的数据格式，实现信号系统与总控数据库节点之间的信息交换。

#### **数据处理**

数据发送时，在未接收到对方消息或者通信中断时，则[上次接收对方序列号]需填写为默认值（0xFFFFFFFF）。

若接收序列号大于上次接收序列号则认为该数据为新数据，则认为有效。若接收序列号小于等于上次接收序列号，则认为该数据为冗余数据，直接丢弃。

在通信中断后，需要重置上次接收序列号为默认值。

冗余处理后，接收方在接收数据时需对数据时效进行检查。若时效超过最大时效时，直接丢弃该数据。

参考延迟时间 = (接收方当前序列号-报文中[上次接收对方序号])\*0.16。报文时效性约定为1.6s。

接收到对方报文时，需对报文的合法性进行检查（例如值域，CRC）等，检查失败，则认为该周期未收到对方数据。

双方在连续10个周期未接收到对方数据，则直接判断为双方通信中断。

## **司机驾驶模拟台PLC协议**

1. 概述

司机驾驶模拟台PLC采用TCP传输协议与各上位机进行通信。本文档对以太网通信数据、报文格式、数据区进行定义。

2. PLC与上位机通信过程

PLC作为服务器，由上位机发起连接或断开请求。PLC服务建立3不同端口的服务端，由上位机主动连接，连接成功后，PLC会定周期发送数据给上位机。上位机输出至PLC时，仅需发送一次。一个端口仅允许一个上位机（客户端）连接，其他的连接请求会被拒绝。

# **（三）视景系统**

<a id="_Toc233726986"></a>
## **1简介**

<a id="_Toc233726987"></a>
### **1.1 目的**

轨交平台是车辆和信号的综合试验平台，其中的三维建模显示与控制提供了试验平台试验时的前方线路场景，包括场景中的信号机、计轴和信标的布置和位置、道岔的位置和状态、车站和发车指示器、前方轨道的坡道和弯道、运行线路的周边环境，以及前方所见列车，是试验平台不可或缺的一部分。本文件为平台中三维视景的接口文件，供开发和测试人员调试参考，也可供平台使用人员阅读。

<a id="_Toc233726988"></a>
### **1.2 假设**

本接口部分，假设开发者具备的一些基础有：

1. 网络通信的基础知识，如TCP/IP，UDP通信等。
2. 三维模型和驱动的基础了解，主要有模型的面和纹理，驱动的帧率等；
3. 线路场景的组成，主要有线路的纵横断面，车站，信号灯，道岔，计轴，信标等。

<a id="_Toc233726989"></a>
## **2 总体描述**

<a id="_Toc233726990"></a>
### **2.1 限制**

为便于北京地铁9号线仿真平台结构上清晰，调试和用户使用的方便，以及保证前景驱动运行是画面流畅，三维建模显示与控制子系统开发时使用了2个计算机来处理，称为控制计算机和前方视景计算机：

1. 前方视景计算机是三维模型的驱动计算机，因此，要求计算机的硬件具有较强的计算能力，尤其是计算机显示卡要求具有很强的图形处理和渲染的能力，硬件配置越高画面质量越高，流畅性越好；一般建议要求性能参数较高的图形工作站和高性能图形显示卡，并能考虑到一定的扩充能力。
2. 控制计算机实际上功能为2个部分，一是接收ATS发来的道岔、信号机、速度、车辆位置等信息，二是提供用户设置场景的天气、道路障碍等设置界面；要求的计算机配置不高，普通的商业计算机均可。
3. **系统软件配置要求：控制计算机通信的ATS计算机（ES计算机）IP为：18.32.115.27；控制计算机 IP：18.32.115.28，视景计算机 IP限制在同段，建议设置18.32.115.30，和18.32.115.31。**

<a id="_Toc233726991"></a>
### **2.2 常见故障说明**

常见的系统故障一般由网络通信引起，因此：

1. 在硬件上需确保HUB的供电和网线的保联接可靠；
2. 软件上需保证IP的地址的准确性；
3. 设置上需关闭防火墙；
4. ATS发来的道岔、信号机和列车位置如果不在制作的场景范围内，有可能导致场景子系统显示错误或故障。因此，仿真运行时请注意ATS运行的范围。

<a id="_Toc233726992"></a>
## **3 详细接口参数**

1. 视景控制机采用UDP接收ATS发来的信号，其信号包括驾驶时的速度与行驶线路中的位置(区间、距离和方向)；UDP发送包还包含信号机、发车倒计时、道岔状态；以便于仿真平台中ATS运行与显示的状态与场景中显示的状态一致；
2. **仿真平台中ATS与视景控制机间交换间隔为100ms;**
3. **平台中ATS仿真机IP：18.32.115.27,发送端口:8302,接收端口8303；**
4. **视景控制计算机的配置 IP: 18.32.115.28，发送端口:8302,接收端口8303；**
5. 视景控制计算机接收ATS发来的状态后，进行信息处理，如速度的处理和场景可见范围显示的信息内容的发送视景驱动计算机。
