# Daily Git Report

## 概述

从当前 git 仓库中筛选出属于用户的提交记录，按**功能主题/模块**归类总结，生成一份**书面化、结构化**的个人周报 Markdown 文件，输出到 `.daily-report/` 目录。

**核心原则：**
- 不是逐条罗列提交明细，而是按**功能模块**归纳整个时间范围内的完成工作
- 用**表格概览 + 文字说明**的形式呈现，而非简单的要点列表
- 面向小团队协作场景，方便其他成员快速阅读

## 使用方式

```
/daily-git-report
```

执行后主动询问用户的**真实姓名**和**日期范围**，无默认值。

## 执行步骤

### Step 1: 确认用户信息与查询参数

主动询问用户的**真实姓名**（用于报告标题和文件名，如 "张三"），以及**日期范围**。用户必须提供日期范围（如 `2026-07-06..2026-07-10`），无默认值。

```bash
GIT_USER=$(git config user.name)
```

获取 git 用户名用于筛选提交记录，但报告标题和文件名使用用户提供的真实姓名。

### Step 2: 获取提交列表（仅 dev 分支）

只查询 `dev` 分支上的提交，不包含功能分支：

```bash
git log dev --author="$GIT_USER" --since="<start>" --until="<end>" \
  --format="%H||%ai||%s" --reverse
```

输出格式：`commit_hash||2026-07-14 10:30:00 +0800||提交信息`

### Step 3: 按功能主题归类

阅读所有提交信息，根据提交的类型和内容归纳为若干个**功能主题/模块**。每个模块包含相关的多项提交。

例如，以下提交：
- `feat(sim): 支持 evaluation_time 运行时调整`
- `feat(scenario): 移除手动保存，自动保存方案支持重命名`
- `feat(api): 方案保存新增 6 个指标`

应归纳为「方案参数-指标矩阵增强」模块。

### Step 4: 生成 Markdown 周报

**文件名规则：**
- **单日范围**：输出 `.daily-report/<真实姓名>-日报.md`（如 `张三-日报.md`），每次覆盖
- **多日范围**：**只生成一份文件**，输出 `.daily-report/<真实姓名>-第N周周报.md`（如 `张三-第一周周报.md`、`张三-第二周周报.md`）。周次从起始日期推算（7/6~7/10 为第一周，7/11~7/14 为第二周，以此类推）。

**周报格式规范：**

```markdown
# <真实姓名> 第一周周报

**时间：** 2026-07-06 ~ 2026-07-10 **提交数：** N **工作日：** M 天

## 本周工作概览

| 模块 | 类型 | 主要内容 |
|------|------|----------|
| 模块A | feat/docs | 一句话概括本模块完成的工作 |
| 模块B | feat/fix | 一句话概括本模块完成的工作 |
| 模块C | docs | 一句话概括本模块完成的工作 |

---

### 模块A

用自然段落描述本模块做了什么、上下文关联、效果影响。将相关提交整合为一个连贯的叙述，而不是逐条列举。

### 模块B

同理，每个模块一段文字说明。
```

### Step 5: 输出结果

告知用户文件生成路径。如果时间范围内无提交，如实报告。

## 格式详解

### 文件头部

- **标题：** `# <真实姓名> 第N周周报`（单日用 `# <真实姓名> 日报`）
- **元信息行：** 时间范围、提交总数、工作日天数，一行展示
- **表格概览：** 以表格汇总本周所有模块

### 表格列说明

| 列 | 内容 |
|----|------|
| 模块 | 功能模块/主题名称 |
| 类型 | 提交类型标签（feat/fix/docs/refactor/chore 等），多个用 `/` 分隔 |
| 主要内容 | 本模块关键变化的**一句话概括**，不是罗列子项 |

### 文字说明段落

每个模块展开一段**书面化总结**，用自然段落描述：
- **做了什么：** 实现了什么功能、修改了什么逻辑
- **上下文关联：** 如果有多个关联提交，说明它们之间的关系
- **效果/影响：** 如果有明显的效果变化（如性能提升、架构改进），补充说明

### 模块分类建议

- 按 `feat(<scope>)` 中的 scope 自然归组
- 相关文档（docs）归入对应功能模块，除非是独立的文档专项
- `chore` / `refactor` 类在模块文字说明中提及即可
- 零散的小修改归入「其他」模块

## 无提交时的处理

```
# <真实姓名> 日报

**时间：** YYYY-MM-DD **提交数：** 0

当日无提交记录。
```

## 输出目录约定

- 输出目录：`.daily-report/`（项目根目录下）
- 文件名规则：
  - 单日：`<真实姓名>-日报.md`（如 `张三-日报.md`），每次覆盖
  - 多日：`<真实姓名>-第N周周报.md`（如 `张三-第一周周报.md`），周次从起始日期推算
- 如果目录不存在，自动创建
- 如果文件已存在，覆盖
- `.daily-report/` 应加入 `.gitignore`

## 常见问题

**找不到提交：**
- 确认 git 用户配置：`git config user.name`
- 确认 `dev` 分支是否存在：`git branch -a | grep dev`
- 确认日期范围内是否有提交：`git log dev --author="..." --oneline`

**跨分支：**
- 只查 `dev` 分支上的提交。功能分支上的提交不计入周报
